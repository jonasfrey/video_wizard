## Detaillierte Bewertungskriterien

### Projekteingabe (5 Punkte)

#### Vollständigkeit & Qualität

| Punkte | Beschreibung |
|--------|-------------|
| **5** | Alle 7 Elemente vorhanden; mindestens 3 SMART-Ziele vollständig formuliert; mindestens 5 konkrete Anforderungen/Features definiert |
| **4** | Alle 7 Elemente vorhanden; mindestens 2 SMART-Ziele; mindestens 3 Anforderungen definiert |
| **3** | Mindestens 5 von 7 Elementen vorhanden; mindestens 1 messbares Ziel; Anforderungen nur grob umrissen |
| **2** | 3 bis 4 Elemente vorhanden; keine messbaren Ziele; Anforderungen unklar oder fehlend |
| **1** | Nur 1 bis 2 Elemente vorhanden; keine erkennbare Struktur |
| **0** | Nicht abgegeben oder verspätet |

### Kurzdokumentation (20 Punkte)

#### Anforderungen & Zielerreichung

| Punkte | Beschreibung |
|--------|-------------|
| **5** | 100% der Anforderungen aus Projekteingabe einzeln aufgeführt mit Status (erreicht/teilweise/nicht erreicht); Abweichungen mit Begründung dokumentiert |
| **4** | Mindestens 80% der Anforderungen abgeglichen; Abweichungen grösstenteils begründet |
| **3** | Mindestens 50% der Anforderungen erwähnt; Begründungen bei Abweichungen fehlen teilweise |
| **2** | Unter 50% der Anforderungen erwähnt; keine Begründungen bei Abweichungen |
| **1** | Kaum Bezug zur Projekteingabe; keine systematische Überprüfung erkennbar |
| **0** | Abschnitt fehlt vollständig |

#### Technische Dokumentation

| Punkte | Beschreibung |
|--------|-------------|
| **5** | Alle 4 Aspekte (Architektur, Komponenten, Routing, State) dokumentiert; Komponentendiagramm oder Komponentenliste vorhanden; Datenfluss beschrieben |
| **4** | Mindestens 3 von 4 Aspekten dokumentiert; Zusammenhänge erkennbar |
| **3** | Mindestens 2 von 4 Aspekten dokumentiert; oberflächliche Beschreibung |
| **2** | Nur 1 Aspekt dokumentiert; keine Zusammenhänge erkennbar |
| **1** | Technische Beschreibung fast vollständig fehlend; nur Stichworte |
| **0** | Abschnitt fehlt vollständig |

#### Visualisierungen

| Punkte | Beschreibung |

| **4** | Mindestens 1 Wireframe/Mockup und 1 Diagramm vorhanden; verständlich dargestellt |
| **3** | Mindestens 1 Visualisierung vorhanden; wenig aussagekräftig oder unprofessionell |
| **2** | Nur Screenshots ohne Kontext oder Erklärung; keine eigenen Visualisierungen erstellt |
| **1** | Unbrauchbare oder unleserliche Visualisierungen |

| **4** | Mindestens 2 konkrete Herausforderungen mit Lösungsansatz; Reflexion vorhanden |

### Technische Umsetzung (40 Punkte)

#### Codestruktur & README

| Punkte | Beschreibung |
|--------|-------------|
| **5** | Einheitliche Formatierung (Linter konfiguriert); sprechende Variablen und Funktionsnamen; logische Ordnerstruktur; README vollständig: Projektbeschreibung, Voraussetzungen, Installationsschritte, Start-Befehl, Konfiguration externer Abhängigkeiten inkl. Beispiel .env; keine API-Keys im Code |
| **4** | Überwiegend einheitlich; README mit Installationsanleitung vorhanden; externe Abhängigkeiten erwähnt, aber Einrichtung nicht vollständig beschrieben |
| **3** | Grundstruktur erkennbar; inkonsistente Formatierung; README unvollständig; Projekt nicht ohne Rückfragen lauffähig |
| **2** | Chaotische Struktur; README vorhanden aber unbrauchbar; API-Keys oder Secrets im Code gefunden |
| **1** | Keine erkennbare Struktur; README fehlt oder ist leer; Code schwer nachvollziehbar |
| **0** | Kein Code abgegeben oder Code nicht ausführbar |

#### Komponentenarchitektur

| Punkte | Beschreibung |
|--------|-------------|
| **5** | Mindestens 5 sinnvoll aufgeteilte Komponenten; keine Komponente über 300 Zeilen; Props und Events/Emits korrekt für Datenaustausch verwendet; mindestens 1 wiederverwendete Komponente |
| **4** | Mindestens 3 Komponenten; Aufteilung grösstenteils sinnvoll; vereinzelt zu grosse Komponenten (über 400 Zeilen) |
| **3** | Komponenten vorhanden; Aufteilung wenig durchdacht; Vermischung von Verantwortlichkeiten; kaum Wiederverwendung |
| **2** | Nur 1 bis 2 Komponenten; keine sinnvolle Aufteilung; fast alles in einer Datei |
| **1** | Keine Komponentenstruktur erkennbar; monolithischer Code |
| **0** | Keine Komponenten vorhanden |

#### Routing

| Punkte | Beschreibung |
|--------|-------------|
| **5** | Mindestens 4 verschiedene Routes; funktionierende Navigation (Links/Buttons); Deep Links funktionieren (direkter URL-Aufruf); 404-Seite für unbekannte Routen |
| **4** | Mindestens 3 Routes; Navigation funktioniert; Deep Links funktionieren; kein 404-Handling |
| **3** | Mindestens 2 Routes; Navigation funktioniert teilweise; Deep Links funktionieren nicht oder Routing-Fehler |
| **2** | Nur 1 Route; Navigation fehlerhaft; Seiten nicht erreichbar |
| **1** | Routing implementiert aber vollständig defekt |
| **0** | Kein Routing vorhanden |

#### State-Management

| Punkte | Beschreibung |
|--------|-------------|
| **5** | Globaler Store sinnvoll eingesetzt (z.B. Pinia, Vuex, Redux) ODER bewusste, begründete Entscheidung gegen globalen Store in Dokumentation festgehalten; klare Trennung von lokalem und globalem State; mindestens 2 verschiedene State-Bereiche |
| **4** | Store vorhanden; Verwendung grösstenteils sinnvoll; vereinzelt globaler State für rein lokale Daten |
| **3** | Kein zentraler Store ohne Begründung; State-Verwaltung unübersichtlich oder inkonsistent |
| **2** | State-Verwaltung chaotisch; Daten gehen bei Navigation verloren; keine Konsistenz |
| **1** | Kein erkennbares State-Management; Anwendung verhält sich unvorhersehbar |
| **0** | State-Management nicht vorhanden; Anwendung nicht funktionsfähig |

#### Funktionalität

| Punkte | Beschreibung |
|--------|-------------|
| **5** | 100% der in Projekteingabe definierten Hauptfeatures funktionieren fehlerfrei |
| **4** | Mindestens 80% der Hauptfeatures funktionieren; kleinere Bugs, die Nutzung nicht wesentlich beeinträchtigen |
| **3** | Mindestens 60% der Hauptfeatures funktionieren; grössere Bugs oder fehlende Kernfunktionen |
| **2** | 30 bis 60% der Hauptfeatures funktionieren; kritische Bugs vorhanden |
| **1** | Unter 30% der Hauptfeatures funktionieren; Anwendung kaum nutzbar |
| **0** | Anwendung startet nicht oder keine Features implementiert |

#### UI/UX Design

| Punkte | Beschreibung |
|--------|-------------|
| **5** | Konsistentes Farbschema (maximal 3 bis 4 Hauptfarben); einheitliche Abstände und Schriftgrössen; klare visuelle Hierarchie; intuitive Navigation (maximal 2 Klicks zu Hauptfunktionen); Ladezustände und Fehlermeldungen vorhanden |
| **4** | Grösstenteils konsistentes Design; kleinere Inkonsistenzen bei Abständen oder Farben; Navigation verständlich |
| **3** | Grundlegendes Design vorhanden; inkonsistente Gestaltung; Navigation teilweise verwirrend; keine Ladezustände |
| **2** | Kein erkennbares Designkonzept; verwirrende Navigation; keine Benutzerführung |
| **1** | Unbenutzbare Oberfläche; grundlegende UI-Elemente fehlen oder sind defekt |
| **0** | Keine UI vorhanden oder Anwendung nicht bedienbar |

#### Responsive Design

| Punkte | Beschreibung |
|--------|-------------|
| **5** | Mobile-First-Ansatz oder optimierte Darstellung für mindestens 3 Breakpoints (Mobile bis 768px, Tablet bis 1024px, Desktop über 1024px); keine horizontalen Scrollbalken; Touch-freundliche Interaktionselemente (mindestens 44px Zielgrösse) |
| **4** | Funktioniert auf Mobile und Desktop; kleinere Darstellungsprobleme auf einem Breakpoint; Inhalte lesbar |
| **3** | Nur für Desktop optimiert oder grössere Layout-Probleme auf Mobile; horizontales Scrollen nötig |
| **2** | Auf Mobile nicht nutzbar; Inhalte abgeschnitten oder überlagert; Buttons nicht klickbar |
| **1** | Layout komplett defekt auf mindestens einem Gerät |
| **0** | Keine Anpassung für verschiedene Bildschirmgrössen; nur auf einem Gerät nutzbar |

#### Git-Workflow

| Punkte | Beschreibung |
|--------|-------------|
| **5** | Sinnvolle Branch-Strategie (mindestens Dev-Branch oder Feature-Branches); alle Gruppenmitglieder haben Commits (sichtbar in Git-Log); mindestens 80% der Commits mit aussagekräftiger Message |
| **4** | Nachvollziehbare History; alle haben mindestens 3 Commits; Commit-Messages meist verständlich |
| **3** | Nur Main-Branch; ungleiche Verteilung (eine Person unter 20% der Commits); nichtssagende Messages (z.B. "fix", "update") |
| **2** | Weniger als 5 Commits total; eine Person hat fast alles commitet; keine sinnvollen Messages |
| **1** | Nur 1 bis 2 Commits; kein nachvollziehbarer Workflow; Code vermutlich am Ende hochgeladen |
| **0** | Kein Git verwendet oder Repository leer/nicht zugänglich |

### Livedemonstration (15 Punkte)

#### Präsentation & Demo

| Punkte | Beschreibung |
|--------|-------------|
| **5** | Alle Hauptfeatures live vorgeführt; klarer roter Faden (Einleitung, Demo, Schluss); Zeitrahmen eingehalten (15 bis 20 Minuten, +/-2 Minuten) |
| **4** | Mindestens 80% der Features gezeigt; erkennbare Struktur; Zeit leicht über oder unterschritten (+/-5 Minuten) |
| **3** | Nur Grundfunktionen gezeigt; kein klarer Aufbau; Zeit stark verfehlt (über 5 Minuten Abweichung) |
| **2** | Unter 50% der Features gezeigt; keine erkennbare Struktur; technische Probleme nicht gelöst |
| **1** | Kaum Features gezeigt; Präsentation unter 5 Minuten oder abgebrochen |
| **0** | Nicht erschienen oder keine Demo möglich |

#### Vortragsweise

| Punkte | Beschreibung |
|--------|-------------|
| **5** | Alle Gruppenmitglieder sprechen mindestens 3 Minuten; freie Rede (kein Ablesen); flüssiger Vortrag ohne längere Pausen; Blickkontakt mit Kamera |
| **4** | Alle beteiligt (mindestens 2 Minuten pro Person); grösstenteils frei; gelegentliches Stocken oder Ablesen |
| **3** | Ungleiche Beteiligung (eine Person unter 1 Minute); stark abgelesen; häufiges Stocken; lange Pausen |
| **2** | Nur eine Person spricht; komplett abgelesen; viele lange Pausen; kein Blickkontakt |
| **1** | Kaum verständlicher Vortrag; keine Vorbereitung erkennbar |
| **0** | Kein Vortrag gehalten |

#### Technisches Verständnis

| Punkte | Beschreibung |
|--------|-------------|
| **5** | Kann alle Architektur-Entscheidungen begründen; beantwortet mindestens 3 Rückfragen korrekt und ausführlich; versteht den gesamten Code der Gruppe |
| **4** | Kann die meisten Entscheidungen erklären; beantwortet mindestens 2 Rückfragen korrekt; kennt den eigenen Code gut |
| **3** | Kann nur oberflächlich erklären; Schwierigkeiten bei Rückfragen; kennt nur Teile des Codes |
| **2** | Kann kaum Fragen beantworten; versteht eigenen Code nicht vollständig |
| **1** | Kann keine Fragen beantworten; kein technisches Verständnis erkennbar |
| **0** | Nicht anwesend oder verweigert Antworten |
